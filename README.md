# 🕷️ CMLABS Web Crawler Service

A production-grade, modular web crawling service built with **Node.js**, **Playwright**, and **Express.js**. Designed to crawl modern websites (SPA, SSR, PWA) and extract fully JavaScript-rendered HTML content.

## ✨ Features

| Feature | Description |
|---------|-------------|
| **JavaScript Rendering** | Full Chromium-based rendering via Playwright — captures content generated by React, Vue, Angular, etc. |
| **Dual Mode** | Works as an Express REST API **and** a CLI tool |
| **Concurrency Control** | Configurable parallel crawling with `p-limit` (default: 3 concurrent) |
| **Retry with Backoff** | Automatic retry up to 3 times with exponential backoff (1s → 2s → 4s) |
| **Fallback Wait Strategy** | Falls back from `networkidle` to `domcontentloaded` + delay for streaming sites |
| **Browser Singleton** | Single Chromium instance reused across all requests — no re-launch overhead |
| **Async Job Queue** | Fire-and-forget crawl with job status polling |
| **Rate Limiting** | In-memory sliding-window rate limiter per IP |
| **Graceful Shutdown** | Handles SIGINT/SIGTERM, closes browser and server cleanly |
| **Structured Logging** | Timestamped, color-coded, level-based logging (INFO/WARN/ERROR/DEBUG) |
| **Input Validation** | URL format validation with detailed error messages |
| **Clean Architecture** | Separation of concerns: config → utils → services → controllers → routes |

## 🏗️ Architecture

```
┌─────────────────────────────────────────────────┐
│                   app.js                         │
│           (CLI mode / Server mode)               │
├──────────────┬──────────────────────────────────┤
│              │                                  │
│   CLI Args   │    Express Server                │
│   Parser     │    ├── Routes                    │
│              │    │   └── crawlRoutes.js         │
│              │    └── Controllers                │
│              │        └── crawlController.js     │
├──────────────┴──────────────────────────────────┤
│              Service Layer                       │
│  ┌──────────────────────────────────────────┐   │
│  │         crawlerService.js                │   │
│  │  • crawlUrl()        — single page       │   │
│  │  • crawlWithRetry()  — with backoff      │   │
│  │  • crawlMultiple()   — batch + p-limit   │   │
│  └──────────────────────────────────────────┘   │
│  ┌──────────────────────────────────────────┐   │
│  │         queueService.js                  │   │
│  │  • enqueue() / markRunning/Completed()   │   │
│  │  • getJob() / listJobs() / cleanup()     │   │
│  └──────────────────────────────────────────┘   │
├─────────────────────────────────────────────────┤
│              Utility Layer                       │
│  logger.js  │  fileHelper.js  │  urlHelper.js    │
├─────────────────────────────────────────────────┤
│              Config Layer                        │
│  browser.js (singleton)  │  index.js (settings)  │
├─────────────────────────────────────────────────┤
│            Playwright (Chromium)                  │
└─────────────────────────────────────────────────┘
```

### Request Flow

1. **Input** → URL(s) arrive via API or CLI
2. **Validation** → `urlHelper.js` checks format, protocol, hostname
3. **Queueing** → URLs are dispatched to `crawlerService` with concurrency limits
4. **Crawling** → Each URL gets its own browser context (isolated cookies/storage)
5. **Rendering** → Playwright navigates, waits for JS, extracts final DOM
6. **Saving** → Rendered HTML is written to `/output` with domain-based filenames
7. **Response** → Results returned with URL, filename, timing, and status

## 🚀 Getting Started

### Prerequisites

- **Node.js** ≥ 18.x
- **npm** ≥ 9.x

### Installation

```bash
# 1. Clone or navigate to the project
cd cmlabs-backend-crawler-test

# 2. Install dependencies
npm install

# 3. Install Playwright Chromium browser
npx playwright install chromium

# 4. Start the server
npm start
```

The server will start on `http://localhost:3000`.

## ⚙️ Configuration

All settings are configurable via environment variables:

| Variable | Default | Description |
|----------|---------|-------------|
| `PORT` | `3000` | HTTP server port |
| `OUTPUT_DIR` | `output` | Directory for saved HTML files |
| `MAX_CONCURRENCY` | `3` | Maximum parallel crawl operations |
| `MAX_RETRIES` | `3` | Retry attempts for failed crawls |
| `REQUEST_TIMEOUT` | `30000` | Navigation timeout per URL (ms) |
| `RETRY_BASE_DELAY` | `1000` | Base delay for exponential backoff (ms) |
| `FALLBACK_WAIT` | `3000` | Wait time after DOM load fallback (ms) |
| `HEADLESS` | `true` | Run browser in headless mode |
| `VIEWPORT_WIDTH` | `1920` | Browser viewport width |
| `VIEWPORT_HEIGHT` | `1080` | Browser viewport height |
| `RATE_LIMIT_WINDOW_MS` | `60000` | Rate limit window duration (ms) |
| `RATE_LIMIT_MAX_REQUESTS` | `10` | Max requests per window per IP |

**Example:**
```bash
MAX_CONCURRENCY=5 MAX_RETRIES=2 node src/app.js
```

## 🌍 SPA / SSR / PWA Handling

### The Problem

Traditional HTTP clients (`axios`, `fetch`, `curl`) download only the initial HTML — which for modern apps is often just an empty `<div id="root"></div>`. The actual content is rendered by JavaScript after page load.

### Our Solution

| Strategy | How It Works |
|----------|-------------|
| **Playwright Chromium** | Launches a real browser that executes JavaScript, loads async data, and renders the full DOM |
| **`networkidle` wait** | Waits until there are no network requests for 500ms — indicates the page has finished loading data |
| **Fallback strategy** | If `networkidle` times out (streaming sites, WebSockets), falls back to `domcontentloaded` + a configurable delay |
| **Context isolation** | Each URL gets its own browser context — separate cookies, storage, and cache |
| **Realistic viewport** | 1920×1080 viewport and real Chrome user-agent to prevent bot detection |

### Website Type Support

| Type | Example | Handled? |
|------|---------|----------|
| **SPA** (Single Page App) | React, Vue, Angular apps | ✅ Full JS rendering |
| **SSR** (Server-Side Rendered) | Next.js, Nuxt.js | ✅ Captures hydrated content |
| **PWA** (Progressive Web App) | Service worker apps | ✅ Chromium executes service workers |
| **Static sites** | HTML/CSS only | ✅ Works out of the box |

## ⚡ Performance Considerations

| Optimization | Impact |
|-------------|--------|
| **Browser singleton** | Eliminates ~2s startup per crawl. One browser, many contexts. |
| **Concurrency control** | Prevents OOM from too many simultaneous tabs. Default: 3 parallel. |
| **Context (not page) reuse** | Contexts are lightweight but provide full isolation. |
| **Resource measurement** | Every crawl reports exact duration in ms for monitoring. |
| **Graceful shutdown** | Ensures browser resources are freed on process exit. |
| **Rate limiting** | Prevents API abuse and resource exhaustion from external callers. |
| **Memory cleanup** | Contexts and pages are closed after each crawl, preventing memory leaks. |

## 📄 Example Output

After crawling `https://cmlabs.co`, the file `output/cmlabs.co_1682323200000.html` contains the **fully rendered** HTML, including:

- All dynamically loaded content (hero sections, pricing tables, etc.)
- Content injected by JavaScript frameworks
- Resolved lazy-loaded elements
- Complete DOM tree as seen by a real browser

This is in contrast to a raw HTTP response which would only contain the initial server-rendered shell.
